Mission

私たちのミッション

TED 開発チームが存在する理由と、日々の仕事で大切にしていること。

「信頼できるソフトウェアを、
持続可能な方法で届け続ける」

TED 開発チームは、顧客・社内・社会に対して価値あるソフトウェアを提供します。

その開発プロセスそのものが信頼に値するものであることを、ルールと文化で保証します。

Vision

目指す姿

2〜3年後に実現したいゴールイメージです。

再現性のある開発

誰がどのプロジェクトに入っても、同じ品質・スピードで動き出せる開発環境と文化を作る。

セキュリティファースト

最初からガードレールが整い、「後でセキュリティを考える」が構造的に起きないチームを作る。

AI 活用の先進チーム

AI ツールを安全・効果的に活用し、開発生産性を高め続ける先進的なチームを目指す。

自律的なチーム

ルールが内面化されており、管理者に都度確認しなくても高品質なアウトプットが出せる。

Principles

開発原則

意思決定・設計・レビュー時に立ち返る、チームの羅針盤です。

01
シンプルさを優先する
複雑な解決策より、単純で明快な解決策を選ぶ。「今必要なもの」だけを作る。
02
セキュリティはオプションではない
セキュリティ対策は後付けではなく、設計の最初から組み込む。
03
透明性・記録を大切にする
口頭で決めたことは記録に残す。「なぜそうしたか」の文脈を守る。
04
小さく動かして確認する
大きな変更を一度に入れない。小さい PR・小さいリリースを繰り返す。
05
レビューは品質への投資
コードレビューは「設計の意図・影響範囲・代替案」まで考える場。
06
自動化できることは自動化する
テスト・Lint・セキュリティスキャンは CI/CD で自動化し、人間はより重要な判断に集中する。
07
FD-E と FD-C は対等なパートナー
技術（FD-E）と顧客・事業（FD-C）は上下関係ではなく、相互補完の関係。技術的実現可能性と事業的価値の両方を考慮して意思決定する。
How to Join

参加申請の方法

Pull Request で参加申請を行います。PR がマージされると Organization への招待が届きます。

1

リポジトリを Fork する

このリポジトリ（ted-occ/web-welcome）を自分のアカウントに Fork してください。

2

自己紹介ファイルを作成する

members/ に自分のユーザー名でファイルを作成します。

members/your-github-username.md
3

テンプレートに沿って記入する

members/_template.md をコピーして、自己紹介・希望ロール・参加動機を記入してください。

4

Pull Request を送る

main ブランチに向けて PR を作成してください。タイトルは以下の形式で。

Join request: @your-github-username
5

招待を待つ

TED-Administrator が PR を確認し、問題がなければマージ後に Organization への招待が GitHub 登録メールアドレスに届きます。
